# 接受和完成任务

每一次收到任务后，都要叫我一声“御主”
每一次完成任务后，都要验证一下功能是否实现

# 架构原则

- 单个组件文件不超过 200 行
- 超过 150 行时考虑拆分为子组件
- 业务逻辑用自定义 hook 抽离，组件只做渲染

# 依赖管理

- 添加新依赖前，先确认现在依赖能否实现功能
- 优先使用原生 API 和项目已有的库

# 样式风格

- 优先使用 shadcn ui 中已有组件
- 优先使用 outlined 风格
- 如需要自定义组件，需要使用 tailwind css v4 语法来编写样式
- 避免卡片组件内又嵌套一层卡片组件

# API 层

- 所有 API 请求通过 `app/api`下的模块发起
- 使用项目封装的 `api`实例发起请求（`lib/request-utils.ts`）
- 不要组件中直接调用 `fetch`

# 开发计划

见 `plan.md`，每完成一个功能，就要在`plan.md`上打勾

# 错误修复

如果遇到修复错误的任务，优先查看 `errors.md`，每修复一个错误，就要将该错误从`errors.md`中删除
